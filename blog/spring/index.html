Spring과 Spring Boot

스프링부트에 대해서 알아봅시다.

시작하기에 앞서

SpringFrameWork를 사용한지는 약 1년정도 되었지만, 개념에 대해서 다시 알아보고자 정리해봅니다.

스프링이란?

스프링은 Java Enterprise Develop을 편하게 도와주는 오픈소스 경량급 프레임워크 입니다.

스프링의 특징

  • IOC (Invertion Of Control) 제어 역전
  • DI (Dependecy Injection) 의존성 주입
  • AOP (Aspect-Oriented Programming) 관점 지향 프로그래밍

IOC ?

객체 생성 부터 Life Cycle까지 Object에 대한 제어권이 개발자가 아닌 FrameWork 에 존재한다는 의미입니다.

예를 들어보자면,

과거에는 의존관계를 개발자가 직접 제어했습니다.

@Repository
public class TestRepository{
}

@Service
public class TestService{
  TestRepository testRepository = new TestRepository();
}

하지만, IOC의 특징을 설명하면 다음과 같이 작성됩니다.

@Service
public class TestService {
  TestRepository testRepository;

  @Autowired
  public TestService(TestRepository testRepository) {
    this.testRepository = testRepository;
  }
}

DI ?

의존성이 있는 객체를 직접 생성하거나 제어하는 것이 아니라, 필요한 객체를 외부에서 가져온다는 의미이다.

public class Exam {

  private Score score;

  public Exam() {
    this.score = new Score();
  }

  public void ShowScore() {
    this.score.showScore();
  }
}
  • Exam 객체에서 ShowScore()가 수행되려면 Socre 클래스가 필요합니다.
  • 즉, Exam클래스는 Score클래스의 의존성을 지니고 있습니다.

위의 설명은 결합도가 매우 높은 상태입니다.
Score 클래스가 수정되면 Exam 클래스도 수정이 되어야 하기 때문입니다.

DI를 구현하는 3가지 방법

  • Filed에 @Autowired
  • Setter에 @Autowired
  • Constructor에 @Autowired

Filed에 @Autowired 

public class Exam {

  @Autowired
  private Score score;

  public void ShoScore() {
    this.score.ShowScore();
  }
}
  • 가독성이 높고 편리합니다.
  • final을 선언할 수 없고, 불변성이 없으며, 외부에서 주입할 수 있는 방법이 없습니다.

Setter에 @Autowired 

public class Exam {

    private Score score;

    @Autowired
    public void setExam(Score score) {
        this.score = score;
    }

    public void ShowScore() {
        this.score.ShowScore();
    }

}
  • Runtime시에 의존성을 주입할 수 있는 낮은 결합성의 특징을 가집니다.
  • 하지만 NPE 발생 우려가 있습니다.

Constructor에 @Autowired 

public class Exam {

    private Score score;

    @Autowired
    public Exam(Score score) {
        this.score = score;
    }

    public void ShowScore() {
        this.score.ShowScore();
    }
}​
  • Null을 주입하지 않는 이상 NPE 발생 우려는 없습니다.
  • 의존관계 주입 없이는 Controller의 객체를 생성할 수 없습니다. [NPE 방지]
  • final 선언 가능
  • 순한 의존성
  • Test 코드 작성시 용이합니다.

AOP ?

OOP를 보완하는 프로그래밍 기법이다.

공통 기능들을 모듈화 하여, 필요할 때 연결하여 유지보수 또는 재사용에 용이함
모듈화란, 공통된 로직이나 기능을 하나의 단위로 묶는것을 일컫는다.

공통된 부분을 쉽게 프로그래밍

  • 각 클래스에 공통된 부분들이 있습니다.
  • 만약, Class A의 주황색 부분을 수정한다면, Class B와 C의 주황색도 수정해야 하는 번거로움이 있습니다.
  • 흩어진 관심사 라고 표현합니다.

위와 같은 상황을 줄이고자 하는 개념이 AOP 입니다.

AOP의 경우 중요한 개념이기 때문에 추후에 더 깊게 다루도록 하겠습니다.
