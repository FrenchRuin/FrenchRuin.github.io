혼자 공부하기 위해 모아온 CS 지식 모음집

프로세스 (Process)

메모리상 실행중인 프로그램을 말하며 디스크로부터 메모리에적재되 CPUT 할당을 받을 수 있는것.
프로세스는 최소 하나의 스레드를 보유하며 별도의 주소공간을 독립적으로 할당받음.

스레드 (Thread)

프로세스 내에 하나의 실행 단위
프로세스내에서 스레드는 자원을 공유할 수 있고, 하나의 프로세스에 다수의 스레드를 실행 단위로 구분한것을 멀티스레드라 함.

멀티프로세스가 아닌 멀티스레드로 처리하는 이유?
멀티스레드로 작업시 멀티 프로세스보다 공유하는 통신 비용이 적고 프로세스 생성 후 자원을 할당하는 시스템 콜이 감소 할 수 있기때문!
대신 공유자원으로 인한 동기화를 고려해야함.

교착상태 (DeadLock)

프로세스가 자원을 얻지 못해 다음 처리를 못하는 상태에 빠진 것.
한정된 자원보다 더 사용하려고 할때 발생하며 교착상태 발생 필수 조건은 아래와 같다.

  • 상호배제 : 필요로 하는 자원에 대해 배타적 통제권 요구
  • 점유대기 : 할당된 자원을 가진 상태에서 다른 자원 기다림
  • 비선점 : 어떤 자원의 사용이 끝날때까지 해당 자원을 뺏을 수 없음
  • 순환대기 : 순환적으로 다음 프로세스가 요구하는 자원을 가지고 있음

RESTful API

API 설계의 중심엔 자원이 있고, HTTP Method를 통해 자원을 처리하도록 설계하는 방식

  • 리소스와 행위를 명시적이고 직관적으로 분리
  • GET, POST, PUT, PATCH, DELETE로 표현
  • Header와 Body를 명확히 분리해서 사용 [ Entity 내용은 Body에 담아서 요청 ]
  • API 버전 관리

장점

  • 멀티프롤랫폼 지원 및 연동에 유리
  • 기존 웹 HTTP 인프라를 그대로 사용

단점

  • 메소드가 4가지만 존재함
  • HTTP 통신 프로토콜에 대해서만 지원

HTTP 프로토콜 (Protocol)

서버/클라이언트 모델을 따라 데이터를 주고 받기 위한 프로토콜
TCP/IP 위에서 동작하며 Method, Header, Body, Path 등으로 Request가 구성
도청, 변조가 가능하여 보안적으로 안전한 Https를 사용하기도 함

Response Code

  • 100번대 : 정보확인
  • 200번대 : 성공
    • 200 - 서버 요청 처리 성공
    • 201 - 요청 접수 후 새 리소스 작성
  • 300번대 : 리다이렉트
    • 301 - 요청 페이지가 새 위치로 영구적 이동
    • 302 - 임시이동 요청자
  • 400번대 : 클라이언트 오류
    • 400 - 클라이언트 오류
    • 401 - 권한 없음
    • 404 - 서버에 요청한 리소스를 찾을 수 없음
  • 500번대 : 서버측 오류

HTTP vs HTTPS
HTTP 와 HTTPS의 차이점은 HTTP통신하는 소켓부분을 SSL이라는 프로토콜 존재 여부

  • HTTP 동작순서 : TCP --> HTTP
  • HTTPS 동작순서 : TCP --> SSL --> HTTP
  • SSL을 사용하지 않는 경우는 암호화로 인해 속도저하가 발생하기 떄문

SSL ( Secure Socket Layer )

보안 프로토콜을 통해 클라이언트와 서버가 통신
전송된 모든 데이터가 비공계로 유지 (TLS - 전송계층보안)
SSL 암호화를 통해 연결을 보호하고 데이터의 변조를 방지하는 역할

웹 통신의 흐름 _ 1

  1. 주소창에 URL을 입력 후 Enter를 누르면 서버에 요청이 전송
  2. 해당 페이지 PATH에 존재하는 자원들을 응답으로 보내줌
  3. 브라우저는 자우너이 담긴 HTML과 CSS를 W3C 명세에 따라 해석
  4. 해석한 렌더링 엔진은 HTML 파싱작업을 시작하며 DOM 트리를 구축
  5. CSS 파싱 작업을 시작하며 CSS가 담긴정보를 스타일 구조체로 생성
  6. HTML과 CSS를 연결하여 렌더트리를 만들고 시각적 요소를 포함한 형태로 출력을 준비
  7. 화면 배치를 시작하고 출력 진행
  8. 이때 브라우저 화면에 표시될때, "배치, 렌더링 과정"은 기다리는 동시에 일부분 먼저 진행하고 화면에 표시

웹 통신의 흐름 _ 2

  1. 브라우저는 DNS 서버로 가서 웹사이트의 진짜 서버주소를 찾음
  2. 브라우저는 서버에게 웹사이트 사본을 클라이언트에게 보내달라는 HTTP 요청 메세지를 서버로 전송
    • 메세지 및 클라이언트 사이에 전송된 모든 데이터는 TCP/IP 연결
  3. HTTP 요청을 받은 서버는 클라이언트 요청을 승인 후 메세지를 클라이언트에게 전송
  4. 서버는 응답해줄 파일들을 데이터 패킷이라는 작은 덩어리로 브라우저에 전송
  5. 브라우저는 패킷들을 웹사이트로 조립 후 사용자에게 보여줌

GET

  • HEADER 부분에 URL이 담겨 전송
  • URL 공간에 데이터가 담겨 전송되기전에 보안에 취약하고 데이터 크기도 제한적
  • 브라우저에서 CACHING 가능

POST

  • BODY 부분에 데이터가 담겨서 전송
  • 보안면에서 GET보다 뛰어남

GET vs POST

GET은 Select의 개념에 가까워 데이터의 리스트 또는 Object 1개를 조회
POST는 서버의 상태나 값이 변경되는 작업을 요청 가능

쿠키 ( Cookie )

클라이언트 (브라우저) 로컬에 저장되는 키:밸류 데이터 파일
유효시간을 명시 가능하며, 브라우저가 종료되어도 인증은 유지
최대 300개까지 저장이 가능하며 도메인당 20개의 값을 할당 가능
자동로그인, 팝업, 더 이상 이창을 보지않음, 장바구니, 아이디 자동저장등 기능에 사용

세션 ( Session )

쿠키를 기반하지만, 서버측에서 관리되기 때문에 웹에서만 사용 가능
클라이언트가 요청시 서버의 엔진이 클라이언트에게 유일한 세션 ID를 부여
브라우저가 종료되면 인증상태가 종료되며, 서버측에 관리되어 보안은 좋지만 사용자가 많을 수록 서버메모리에 있어 비효율적

쿠키와 세션을 사용하는 이유 ?
정보가 유지되지 않으면, 매번 페이지를 이동할 때마다 인증을 다시하거나, 상품을 선택했는데 구매 페이지에서 선택한 상품의 정보가 없는 일이 발생
큰 차이점은, 쿠키는 로컬PC에 저장하고, 세션은 서버에 저장

쿠키 동작순서

  1. 클라이언트 페이지 요청
  2. 웹서버 쿠키생성
  3. 쿠키 정보를 담아 HTTP 응답을 줄 때 클라이언트에게 전달
  4. 클라이언트는 받은 쿠키정보를 로컬 PC에 저장했다가 서버에 다시 요청시 쿠키와 함께 전송
  5. 서버는 쿠키가 있는 경우 요청페이지와 함께 쿠키를 다시 전송

세션 동작 순서

  1. 클라이언트 페이지 요청
  2. 서버는 클라이언트의 REQUEST_HEADER 필드인 쿠키를 확인하고 클라이언트가 세션ID를 보냈는지 확인
  3. 세션 ID가 존재하지 않으면 세션ID 생성 후 클라이언트에게 전달
  4. 서버에서 클라이언트로 돌려준 세션 ID를 쿠키를 사용해 서버에 저장 [ JSessionID ]
  5. 클라이언트는 재 접속시 쿠키를 이용해 세션ID를 서버에 전달
